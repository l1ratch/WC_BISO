# ✅ **Вариант №1**

## **1. Распределённые события**

В распределённых системах возникает необходимость получать уведомления о событиях, происходящих **асинхронно** и **удалённо**. Существует два подхода:

### **(1) Тесно связанные события**

Издатель напрямую уведомляет подписчика. Недостатки:

*   обе стороны должны работать одновременно;
*   сложно рассылать уведомление сразу нескольким подписчикам;
*   трудно фильтровать и протоколировать события.

### **(2) Слабо связанные события (pub/sub)**

Издатели и подписчики **не взаимодействуют напрямую**.  
Используется **менеджер событий**, который:

*   позволяет подписываться и отписываться;
*   получает событие от издателя и рассылает подписчикам;
*   может находиться на другом компьютере (как и издатели/подписчики).

Событие нередко реализуется вызовом удалённого метода на стороне подписчика.

## **2. Безопасность в распределённых системах**

Для обеспечения безопасности промежуточная среда должна поддерживать **три ключевые функции**:

### **1) Аутентификация**

Проверка подлинности пользователя или компоненты. Может быть:

*   **односторонней** (сервер проверяет клиента),
*   **двусторонней** (клиент также проверяет сервер). 

### **2) Авторизация**

Ограничение доступа на основе **ролей** (role‑based security).  
Разработчик указывает абстрактные роли, а реальные пользователи назначаются администратором.

### **3) Защита передаваемых данных**

Сообщения должны быть **подписаны электронной подписью и зашифрованы** на клиенте и сервере.  
Защита может обеспечиваться транспортным протоколом, промежуточной средой или обоими.  

### **Механизмы управления доступом** (кратко)

В распределённых системах применяются:

*   **матричная модель**;
*   **решётчатая модель (Denning)**;
*   **модель Белла–ЛаПадулы**;
*   **модель потоков информации**;
*   **механизм ядра безопасности и монитор обращений**.

# ✅ **Вариант №2**

## **1. Распределённые транзакции**

**Транзакция** — последовательность операций, которая либо выполняется полностью, либо не выполняется вовсе. Требования ACID:

*   **атомарность**,
*   **согласованность**,
*   **изоляция**,
*   **долговечность**.    

**Распределённая транзакция** охватывает несколько компонент, которые:

*   могут работать с несколькими БД, очередями сообщений, файлами;
*   должны корректно откатывать изменения одновременно.  
    Для этого необходимы:

1.  поддержка распределённых транзакций в промежуточной среде;
2.  использование только тех ресурсов, которые могут участвовать в транзакции. 

### **Подходы к реализации**

#### **(1) Двухфазная фиксация (2PC)**

*   Фаза подготовки (prepare) — компоненты подтверждают готовность.
*   Фаза фиксации — координатор либо фиксирует, либо откатывает всё.  
    Гарантирует атомарность, но медленный и создаёт блокировки. 

#### **(2) Согласованность в конечном счёте / SAGA**

*   Реализуется через обмен событиями между микросервисами.
*   Выполняется как цепочка локальных транзакций.
*   При ошибке выполняются **компенсирующие транзакции**.

## **2. Промежуточные среды в Microsoft .NET Framework**

.NET Framework содержит **четыре основных промежуточных среды**, поддерживающих распределённость:

### **1) MSMQ (Microsoft Message Queuing)**

Обеспечивает обмен сообщениями через очереди (асинхронность, надёжность).   

### **2) Microsoft Enterprise Services / COM+**

Поддерживает удалённые объекты и **распределённые транзакции** внутри локальных сетей.   

### **3) ASP.NET Web Services**

Позволяют выполнять удалённые вызовы на основе стандартов XML и SOAP.   

### **4) .NET Remoting**

Универсальная среда для удалённых объектов (предшественник WCF).

### **WCF (Windows Communication Foundation)**

С версии .NET 3.0 объединяет все предыдущие технологии в единую коммуникационную платформу.

Также платформа включает:

*   CLR,
*   библиотеки классов (Windows Forms, Web Forms, ADO.NET, Web Services и др.),
*   механизмы безопасности и JIT-компиляцию.
